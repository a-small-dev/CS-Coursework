# Tester Reflection

## Overview
This journal reflects on the role and responsibilities of a Tester and how I would apply Scrum events in a real project environment.

## Retrospective
For a product tester, user stories act as the bridge between what the user wants and how the system is tested. When I started thinking about my initial test cases, I tried to break down the user story by thinking who the user is, what they want, and why they want it. This format helped me clarify the purpose behind a feature instead of just focusing on what the system should do. For example, if a user story says, “As a traveler, I want to filter vacation packages by price so that I can stay within my budget,” it immediately tells me what needs to be tested. I know I need to verify that the filter works correctly, that it updates results accurately, and that it handles edge cases like empty results or invalid inputs. Another helpful element is acceptance criteria. When acceptance criteria are included, they basically act like a checklist for testing. This aligns with industry expectations that testers should be able to plan, design, and execute test cases effectively (Hamid & Ikram, 2024).

That said, even with well-written user stories, I think communication with the Product Owner is extremely important. I would go out on a limb and say it’s probably one of the most critical parts of developing accurate test cases. Research shows that communication is one of the most in-demand soft skills in software testing, with over 80% of job postings requiring it (Hamid & Ikram, 2024). This makes sense because testers constantly need clarification and feedback. Without clear communication, it’s easy to misunderstand requirements, which can lead to incomplete or incorrect test coverage.

The importance of communication is also supported by more recent research that highlights how testers must interpret different types of technical artifacts, including user stories, and clearly communicate findings with team members (Hardman et al., 2025). Miscommunication at any stage can negatively impact the testing process, especially in fast-paced Agile environments where requirements can change frequently. The Product Owner plays a key role here by helping clarify expectations, explaining the intent behind features, and confirming priorities. This collaboration makes sure that testers are actually validating that the system meets user needs.

I felt like just overall more detail could have been in the user stories, for example the customers mentioned features like top 5 or 10 destinations or customized lists but they didn’t go into detail on how they would like the lists presented. In the module 4 assignment where we had to do the revised lists to add the slideshow feature is a good example of this. Another example I could note where detail is missing is how exactly do they want top destinations to be defined? Should top destinations be defined by most visited, highest rated, most reviewed, etc. A way to obtain this information would first be prepared and make sure to have the questions/information you need/want answered written down in advance. After preparing these questions, a follow-up meeting can be scheduled with the Product Owner and, if possible, members of the user focus group.

Sample Email:

Hi [Product Owner Name],

I hope you’re doing well. I’m currently working on developing test cases for the travel booking features in our current sprint, and I wanted to reach out for some clarification to make sure my testing aligns with the intended functionality.

I noticed that in the user story related to filtering vacation packages, the acceptance criteria are somewhat limited. Could you provide more details on the following points?

•	What specific filters should be supported (price range, destination, dates, ratings)?

•	Are there any edge cases we should account for, such as no results being returned or invalid user inputs?

•	Are there any performance expectations for how quickly filtered results should load?

•	Should the filters persist if the user navigates away and comes back?

Additionally, for the user story regarding customizable travel preferences, could you clarify which preferences are included and whether users can update them at any time?

Having this information will help me make sure that the test cases cover the requirements. 

Thanks for your time and support!

Anthony Small

Product Tester
